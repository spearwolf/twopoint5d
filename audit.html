Projektportrait

@spearwolf/twopoint5d ist eine TypeScript-Bibliothek, die batch-basiertes 2D-Sprite-Rendering — Billboards, Tilesets, Frame-Animationen, Parallax-Projektionen, Tiled-Maps — in eine three.js / WebGPU-Szene bringt. Das Herzstück ist eine deklarative Vertex-Object-Description: ein Layout-Schema, das je-Instance-Properties auf ein gemeinsames, typisiertes BufferGeometry abbildet, sodass tausende „Sprites“ in einem Draw-Call landen. Die Lib versteht sich bewusst nicht als Sprite-Engine, sondern als Toolkit, das die WebGPU-Zustandshandhabung verbirgt, ohne three.js wegzukapseln.

Das Repo ist ein Nx + pnpm-Workspace mit drei Projekten: der publizierten Kernbibliothek packages/twopoint5d, einem separaten Browser-Integrationstestpaket packages/twopoint5d-testing (@web/test-runner + Playwright) und einer Astro-basierten Showcase-App apps/lookbook, die de-facto-Doku und Demo-Sammlung ist. Ein zusätzlich vorhandenes apps/handbook ist eine Überbleibsel-Verzeichnisstruktur (VitePress wurde entfernt, nur Bilder sind übrig).

Vertex Objects (Core)
Perf-Kern: deklarative Attribut-Layouts, die zu JS-Proxies über geteilte TypedArrays kompiliert werden — die Grundlage für jeden Sprite-Typ.
packages/twopoint5d/src/vertex-objects/
Display & Frame Loop
WebGPURenderer-Wrapper mit Resize-Auto-Sync, Chronometer mit clamping maxDeltaTime, requestAnimationFrame und Fixed-Timestep-Loop für deterministische Simulation.
packages/twopoint5d/src/display/
Stage & Render Pipeline
Stage2D (Scene + Projection + automatische Kamera), StageRenderer mit verschachtelten Render-Targets und TSL-basiertem Post-Processing-Pipeline-Modus.
packages/twopoint5d/src/stage/
Texture Management
TextureAtlas, TileSet, signalize-basierter TextureStore-Cache mit reaktiver Pipeline für Bild-Loader, Atlas-Parsing und FrameBasedAnimations.
packages/twopoint5d/src/texture/
2D Tiled Maps
Tiled-Integration: Map2DLayer, CameraBasedVisibility (View-Frustum-Culling auf XZ-Plane) mit Tile-Pool, Spatial Hash Grid und ChunkQuadTree.
packages/twopoint5d/src/map2d/
Sprite Materials
Fertige ShaderMaterials auf Vertex-Object-Basis: TexturedSprites (Quads/Billboards), AnimatedSprites, TileSprites.
packages/twopoint5d/src/sprites/
Input Controls
PanControl2D (WASD + Pointer mit Multi-Touch-Merging und Cursor-Hide-Strategie).
packages/twopoint5d/src/controls/
Lookbook (Demo-App)
Astro-App mit kuratierten Demos (animated-billboards, animated-sprites, instanced-quads, quadtree-playground, textured-sprites, passes, crosses) — primäre Doku-Quelle.

Executive Summary

Der Code ist überdurchschnittlich gepflegt für ein Solo-Open-Source-Projekt: klare Modulgrenzen via public-api.ts, sehr ausführliche TSDoc auf den zentralen Klassen (Display, StageRenderer, FixedFrameLoop dokumentieren Lifecycles und Trade-Offs sauber), strikt geführte ESM-/NodeNext-Imports, konsequente .js-Suffixe und ein realistischer CI-Pfad (Build → Lint → checkPkgTypes → Test inkl. headless Playwright via xvfb-run).

Die nennenswerten Schwachstellen liegen im Type-System und in Lifecycle-Robustheit: strict: true, aber strictNullChecks: false ist ein bewusst offen gehaltenes Sicherheitsfenster, das in mehreren // @ts-ignore-Stellen und Methoden mit Rückgabewert-Lügen (VertexObjectPool.createVO(), FrameLoop.start()) konkret spürbar wird. Display hat zwei Lifecycle-Klippen (Konstruktor-Pfad mit externem WebGPURenderer setzt #waitForRenderer nie; canvas-Getter wirft nach dispose()). PanControl2D verwendet das deprecated event.keyCode für WASD — bricht auf nicht-lateinischen Keyboard-Layouts.

Tooling-Lücken: keine Coverage-Konfiguration trotz Performance-/Buffer-Code, CI nutzt nick-fields/retry@v3 (3× Wiederholung) — das maskiert Flakes; nur 5 Browser-Integrationstests im twopoint5d-testing-Paket gegenüber 42 Unit-Specs. Aufräum-Backlog: drei Backlog-*.md-Dateien (~110 KB) und das verwaiste apps/handbook/-Verzeichnis liegen weiterhin im Repo-Root. AGENTS.md erwähnt das längst entfernte VitePress, README.md nennt Node 18+ während engines auf 22.13 besteht.

    Optimierungspotenzial

    Bewusst von Findings getrennt: das hier sind Verbesserungen, die kein Defekt sind, aber das Projekt resilienter oder lesbarer machen würden.

    • Coverage in CI — vitest --coverage aktivieren, einen Mindest-Threshold für vertex-objects/ einziehen. Bei einem Buffer-Manipulations-Kern kostet ungetesteter Edge-Case bares Geld in Production-Bugs.
    • ESLint-Regel-Override — '@typescript-eslint/no-empty-object-type': 0 ins TS-Block der eslint.config.mjs, dann fallen die ~10 verstreuten // eslint-disable-next-line-Kommentare weg, die alle exakt dem gleichen Pattern (Declaration-Merging mit EventizedObject) dienen.
    • Pre-Commit-Hook — minimal lint-staged + Husky einziehen; spart CI-Loops bei Tippfehlern. Für ein Solo-Projekt nicht zwingend, aber günstig.
    • .nvmrc / .node-version — eine Zeile, sofortiges Onboarding-Win für Nutzer mit fnm/nvm/volta.
    • Extraktion in Display.resize() — die Resolution-Order der Size-Sources (resize-to-Attribut, Callback, Element) ist in einer einzigen 140-Zeilen-Methode. Drei kleine Helper (resolveSizeFromAttribute, applyBoxModelCorrection, clampToMaxResolution) würden die Lesbarkeit sichtbar heben, ohne das Verhalten zu ändern.
    • README ↔ AGENTS.md ↔ CLAUDE.md synchronisieren — drei Quellen, die teilweise dasselbe sagen (Setup, Stack), teilweise widersprüchlich (Node-Version, VitePress-Erwähnung). Ein generierter Snippet aus package.json + pnpm-workspace.yaml wäre wartungsfreundlicher.
    • noImplicitReturns: true aktivieren — geringer Migrationsaufwand (nur Switch-Statements mit fehlendem Default), enger Sicherheitsgewinn bei Refactorings.
    • Auto-Eviction in TextureStore.parse() — Resources, deren ID im neuen Datenset fehlt und deren refCount === 0, gleich entsorgen statt erst bei clearUnused(). Spart Live-Editing-Speicher in Lookbook-Demos.
